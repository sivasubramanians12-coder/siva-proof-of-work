# Multi-agent architecture — first working setup

Got a working multi-agent setup running on my Mac mini via OpenClaw. Three agents, each with a distinct role, each bound to a specific Telegram context.

**Agents:**
- **BarleyBot** (main) — orchestrator, memory, general ops. Lives in Telegram DM
- **Coffee** — research agent. Bound to a dedicated research Telegram group. Runs on Claude Sonnet 4.6
- **Coder** — coding agent. Claude Opus 4.6. For actual builds

**What broke:**
- `agents.defaults.subagents.model` in the config overrides per-agent model settings. Took 3 hours to find — Coffee kept running on Sonnet despite being configured for Gemini
- Gemini free tier rate-limits instantly on the first agent call. Agents make 8–12 API calls per task. Free tier is not viable
- Direct JSON config edits break silently. Python scripting for all config changes from now on

**Memory architecture (3 layers):**
1. Active session context — ephemeral, lives in the current conversation
2. Daily logs — `memory/YYYY-MM-DD.md` — raw notes written during sessions
3. Long-term `MEMORY.md` — curated, distilled, updated during heartbeats

**Tool notes:**
- OpenClaw heartbeats: runs every 1h, reads `HEARTBEAT.md`, writes daily memory, checks gateway health
- Groq Whisper (`whisper-large-v3-turbo`): voice note → text in <2s. Best inference speed available
- Gemini 2.5 Flash as gateway web search provider — uses existing Google API key, works for sub-agents
- Gateway must run as LaunchAgent, not a manual process — stability difference is significant
