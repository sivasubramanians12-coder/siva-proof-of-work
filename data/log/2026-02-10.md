# Supabase RLS + FastAPI auth — what actually works

Spent most of the day debugging RLS policies on the Geneva pipeline. Field reps should only see their own deals. Managers should see their region. Leadership sees everything.

**What I tried first (wrong):**
Passing the Supabase anon key from the FastAPI backend. This bypasses RLS entirely — the service key has superuser access. Every rep was seeing the full pipeline. Not obvious from the docs.

**What works:**
1. Frontend authenticates via Supabase Auth, receives a JWT
2. JWT is passed in the `Authorization` header to FastAPI
3. FastAPI verifies the JWT signature using Supabase's JWKS endpoint
4. Creates a Supabase client with `headers={"Authorization": f"Bearer {token}"}` — this client respects RLS

**Key insight:** The Supabase client needs to be initialised *per request* with the user's JWT, not once at app startup with the service key. Obvious in retrospect.

**Tool notes:**
- `python-jose` for JWT verification — `jose.jwt.decode()` with the Supabase JWKS
- RLS policies: `auth.uid() = rep_id` for rep-level, `auth.jwt()->>'role' = 'manager'` for managers
- FastAPI dependency injection makes the per-request client pattern clean
