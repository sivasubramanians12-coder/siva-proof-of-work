New project. SiteCapture — a field check measure and customer alignment app for Hillarys India — went from zero to first sprint in one day.

The problem it solves: Hillarys India runs 3 field visits per customer order. The check measure visit — where an installer visits the customer's home to verify measurements and confirm product specifications — had no structured system. Installers used paper notes and WhatsApp photos. CX manually bridged the data. Discrepancies between what was captured at check measure and what arrived at installation were a primary cause of rework.

SiteCapture sits between Ops OS (which manages the order lifecycle) and InSite (the sales/design consultation tool). It connects to Ops OS via JID on the same Supabase project — no sync, no API, same database.

Governance: PRD v1.2 (16 sections, 1270 lines), full `.claude/commands/` runtime (v3), 9-sprint manifest, MANIFEST.md dependency DAG.

Foundation sprint complete: Next.js 16 App Router, TypeScript strict mode, Supabase SSR helpers, shared auth with Ops OS, all 7 `sc_` table migrations with RLS policies, role routing for 4 roles (CX, Installation Coordinator, Installer, Admin).

The PDF parsing work also started: pdfjs-dist integration to extract quote data from InSite-generated PDFs. This was the technically tricky part — CDN loading to avoid server bundle issues, type annotations for text content items.

**Vayu — Home + Profile Redesign**

While SiteCapture was scaffolding, the Vayu origami redesign continued: Home screen and Profile screen brought in line with the design system established yesterday. All 4 main screens now aligned.
